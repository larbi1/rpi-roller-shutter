Section 1 — Project Overview

1.1 Goal

The system controls 4 roller shutters via a Raspberry Pi 5 and a Waveshare RPi Relay Board (B). It provides two independent, simultaneous control channels:

  • Web interface — any phone, tablet, or computer on the local Ethernet or Wi-Fi network opens a browser to http://<rpi-ip>:8081 and controls any shutter from anywhere in the building.
  • Physical buttons — 8 wall-mounted push-buttons (4 UP + 4 DOWN) give instant local control without any device. Inspired by the GCE Electronics X-4VR V2 which uses the same 8-input principle.

Both channels operate simultaneously and independently. Either can be used at any time. A concurrency lock prevents conflicts when both are used at the same moment on the same shutter.

1.2 Scope

  • 4 shutters × 2 relays each = 8 relays (all 8 channels of the Waveshare board used)
  • Web control: Flask REST API on port 8081, dark-theme dashboard, accessible from any device on the network via Ethernet or Wi-Fi
  • Button control: 4 shutters × 2 buttons (UP + DOWN) = 8 buttons total
  • Python daemon using python3-gpiod for event-driven GPIO button monitoring
  • Button behaviour identical to X-4VR V2 Mode 1 “VR – Push Button”
  • Both channels share a single relay controller (shutter_control.sh) with per-shutter locking to prevent race conditions
  • All processes run as systemd services, auto-start on boot

Section 2 — Electrical Roller Shutter Specifications

2.1 Motor Type

Standard European roller shutter motors: single-phase AC, 230 V 50 Hz. Three conductors:

  • Montée / UP — phase energised for upward travel
  • Descente / DOWN — phase energised for downward travel
  • Neutral — common return; some motors add PE (protective earth)

Active direction is determined by which phase conductor is energised; the other floats.

CRITICAL: UP and DOWN conductors must NEVER be energised simultaneously — this would short two phase legs, destroying the motor or tripping the breaker.

2.2 Electrical Ratings

ParameterValue
Supply voltage230 V AC ±10%, 50 Hz
Motor power30 W – 200 W (depends on shutter size)
Running current0.15 A – 0.9 A at 230 V
Inrush current3–8× running current, first 100–300 ms
End stopsMechanical or electronic (built into motor)
Thermal protectionBuilt into motor (auto-reset after cool-down)
Power factor0.8 – 0.95

2.3 Relay Requirements

Waveshare RPi Relay Board (B) relay contact rating: 10 A @ 250 V AC (resistive load). This is well above motor running current (max ~0.9 A). Inrush current (up to ~7 A for 100 ms) is within relay contact rating. Relay switching life: ~100,000 operations (electromechanical).

Note: For inductive motor loads, an RC snubber across relay contacts is recommended to reduce arc damage — optional for low-power motors but good practice.

2.4 Hardware Interlock Requirement

Software interlock: shutter_control.sh enforces a 0.5 s delay between releasing one relay direction and engaging the opposite.

Optional: mechanical interlock relay for belt-and-suspenders hardware safety — not planned for this project but recommended for high-cycle installations.

Wiring

Both models wire identically. For each contact:

  • Contact leg A → RPi GND (any GND pin: 6, 9, 14, 20, 25, 30, 34, or 39)
  • Contact leg B → RPi GPIO input pin (see Section 4.2)

Internal pull-up enabled by button_daemon.py: idle = HIGH (3.3 V), pressed = LOW (0 V). One shared GND rail for all 8 contacts is acceptable.

5.2 Why python3-gpiod for Buttons?

  • Event-driven: kernel captures GPIO edges via hardware interrupt — zero missed presses
  • Efficient: Python only processes the kernel event queue — GIL and GC are irrelevant for human-speed input
  • No polling loop — zero CPU usage when idle, accurate press timing
  • Consistent with libgpiod already used by shutter_control.sh
  • Bash is unsuitable: requires spawning a gpioget subprocess every 50 ms — unreliable, high overhead, can miss short presses

Section 6 — Button Daemon Architecture (button_daemon.py)

6.1 Design Principles

  • Event-driven via python3-gpiod edge monitoring (falling + rising edges)
  • Linux kernel captures button edges via hardware interrupt — microsecond precision
  • Python reads from kernel event queue — no missed presses, zero CPU when idle
  • Completely independent from web server — both call shutter_control.sh
  • Per-shutter flock lock files prevent race conditions with web server
  • Auto-detects gpiod API version at startup — no manual configuration required

6.2 gpiod Version Compatibility

VersionEvent loop methodAPI used
v2.x (current) request_lines() + wait_edge_events() gpiod.LineSettings(edge_detection=Edge.BOTH, bias=Bias.PULL_UP)
v1.x (legacy) get_line() + select.select() on event_get_fd() LINE_REQ_EV_BOTH_EDGES + LINE_REQ_FLAG_BIAS_PULL_UP

Version detected at startup via gpiod.__version__. Both paths use time.monotonic() for press duration measurement (avoids timestamp API differences).

6.3 Button Behaviour (Mode 1 — VR Push Button, inspired by X-4VR V2)

Press typeShutter stateAction
Short press UP (< 3 s)stoppedshutter_control.sh up N
Short press UP (< 3 s)moving upshutter_control.sh stop N
Short press UP (< 3 s)moving downshutter_control.sh stop N
Short press DOWN (< 3 s)stoppedshutter_control.sh down N
Short press DOWN (< 3 s)moving downshutter_control.sh stop N

Long press any (≥ 3 s)anyshutter_control.sh stop all
Long press triggers on button release (after ≥3 s held). User can abort by releasing early if they held for less than 3 s.

6.4 Event Processing Flow

  1. Kernel detects falling edge (button press) → delivers event to Python via gpiod fd
  2. Record press timestamp and button identity (GPIO pin number)
  3. Continue monitoring for rising edge (button release) on same pin
  4. Kernel detects rising edge → Python calculates duration = release_time − press_time
  5. Apply debounce filter: if duration < 50 ms → discard (mechanical bounce / noise)
  6. If duration < 3000 ms → short press logic (read state file, choose up/stop/down)
  7. If duration ≥ 3000 ms → long press logic (all stop)
  8. Acquire per-shutter lock file (non-blocking — drop command if locked)
  9. subprocess.run(["shutter_control.sh", action, shutter_number])
  10. Release lock file

6.7 Startup Sequence

  1. Verify /usr/local/bin/shutter_control.sh is accessible and executable
  2. Detect GPIO chip via gpiodetect (prefers rp1-gpio chip on RPi 5)
  3. Detect gpiod API version via gpiod.__version__ → select v1 or v2 event loop
  4. Open 8 GPIO lines with BIAS_PULL_UP and EDGE_BOTH configuration
  5. Log startup: chip, gpiod version, pins configured, daemon ready
  6. Enter blocking event loop — zero CPU usage when idle
  7. On SIGTERM or SIGINT: release all GPIO lines, flush log, exit cleanly

Section 7 — Safety Design

7.1 Electrical Isolation

Button wiring (3.3 V / GND, signal level) is completely separate from mains wiring (230 V AC).
NEVER connect button terminals to the relay board mains-side screw terminals.
Route button cables in separate conduit or use clearly labelled cable.
If buttons are in the same wall box as mains: use a separate compartment or earth-screened cable.

7.2 Software Interlock (existing, preserved)

The 0.5 s motor direction-change delay is enforced inside shutter_control.sh. button_daemon.py NEVER calls GPIO directly — always calls shutter_control.sh. The interlock is therefore always active regardless of whether commands originate from buttons or web.

7.3 Race Condition Prevention

  • Per-shutter lock files ensure only one command executes at a time per shutter.
  • If web server is commanding SH1 when a button is pressed for SH1: button command is silently dropped (safe — shutter is already being controlled).
  • Different shutters can be commanded simultaneously — they are physically independent.

7.4 Long Press Threshold

3 seconds minimum, triggers on release. Always maps to ALL STOP — safest possible global action. Prevents accidental global commands from short presses.

7.5 Boot Safety

  • On RPi reboot: GPIO reconfigured fresh at daemon startup, relay daemons not running, state files absent (treated as “stopped”).
  • Relays default to de-energised (OFF) at boot — shutters stay at last physical position.
  • No automatic shutter movement on boot.
  • button_daemon.py starts after shutter-web.service via systemd ordering.

7.6 Simultaneous Button Presses

ScenarioResult
Same shutter, both UP and DOWN pressedLock file drops the second command — safe
Different shutters pressed simultaneouslyFully independent, concurrent commands allowed
